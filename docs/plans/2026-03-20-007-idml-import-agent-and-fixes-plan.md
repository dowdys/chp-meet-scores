# Plan 007: IDML Import Agent & Fixes

## Source
Process log analysis from 2026 Nevada State Championships IDML import session. User edited two IDML files in InDesign (letter and legal backs) and returned them. Agent struggled to handle the import correctly.

## Core Problems

### Problem 1: IDML without metadata goes to wrong folder
**Log line 2102**: "No embedded metadata found in IDML" → output goes to "IDML Import" folder instead of the correct meet folder. The first IDML file ("NV 2026 2-10.idml") had no CHP_METADATA, so no order forms or gym highlights were generated.

**Root cause**: CHP_METADATA is only embedded in IDML files generated by the app. If the user renames the file or it was generated before metadata embedding was implemented, the metadata is lost.

**Fix**:
- When metadata is missing, the agent should use `set_output_name` to target the correct meet folder BEFORE calling `import_idml`
- The `import_idml` tool should accept optional `state` and `meet_name` parameters as fallback when metadata is missing
- The IDML generator should ALWAYS embed metadata — verify this is happening for all IDML outputs including legal-size

### Problem 2: Gym highlights and order forms don't reflect IDML changes
**Log lines 2219-2223**: User says "the gym highlights and order forms still have the old back instead of the back that I gave you." The agent responded by asking the user what changes they made — this is wrong.

**Root cause**: After IDML import, `gym_highlights.pdf` and `order_forms.pdf` are regenerated from the database using code-template styling (hardcoded fonts, sizes, colors). The user's InDesign edits (which may include font changes, color changes, spacing adjustments) are NOT extracted from the IDML and applied.

**The real workflow should be**:
1. Import IDML → produces updated `back_of_shirt.pdf` from the designer's IDML
2. The `gym_highlights.pdf` and `order_forms.pdf` should use the IMPORTED back_of_shirt.pdf pages as their "back" reference images, NOT regenerate from code
3. Alternatively: extract styling parameters from the IDML (font, size, color, spacing) and pass them to the regeneration

**Fix approach A (simplest)**: The gym_highlights and order_forms already embed the back_of_shirt page as a background/reference. After IDML import, just regenerate them — they should pick up the new back_of_shirt.pdf automatically. If they don't, the regeneration code needs to read the current back_of_shirt.pdf rather than re-rendering from data.

**Fix approach B (extract styling)**: Parse the IDML XML to extract font family, font sizes, accent color, and line spacing. Pass these as parameters to regenerate_output. This is more robust but complex.

### Problem 3: Two IDML files for one meet (letter + legal)
The user returned TWO IDML files:
- "NV 2026 2-10.idml" (letter size, Levels 2-10) — no metadata
- "NV 2026 XCEL 8.5X14 (1).idml" (legal size, Xcel) — has metadata

The agent imported them sequentially but the first one went to the wrong folder. The correct workflow for multi-back meets:
1. Import letter-size IDML to the correct meet folder
2. Import legal-size IDML to the same meet folder
3. Combine if needed (or keep as separate files)
4. Regenerate order forms using the CORRECT back_of_shirt.pdf
5. Regenerate gym highlights split by level groups (letter gets Levels 2-10, legal gets Xcel)

### Problem 4: Agent should NOT try to manually replicate InDesign changes
**Log line 2311-2323**: Agent asked "Could you tell me what specific style changes you made in InDesign?" This is fundamentally wrong. The whole point of IDML import is that the designer's changes are IN the IDML file. The agent should never try to manually recreate what the designer did.

### Problem 5: Discovery phase wastes iterations browsing MSO
**Log iterations 1-20**: Agent spent ~20 iterations browsing MSO website instead of using `mso_extract` tool directly. It knew the meet ID (34775) early but didn't extract until much later.

**Fix**: The discovery phase prompt should emphasize: once you have a meet ID, use `mso_extract` immediately. Don't browse the MSO website looking for data.

### Problem 6: IDML metadata not always embedded
The user said "typically we put metadata in the idml files that identifies which meet it is tied to and I'm not sure if that is always happening." Need to verify that ALL IDML generation paths embed CHP_METADATA.
